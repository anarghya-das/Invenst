"Tab completion is a common feature in command-line interfaces. In bash, for example, if I type: `$> ec\t` The command line gets replaced by: `$> echo ` This is relatively easy to implement, as the semantics of command-line expressions are positional: The first field is always a command, the second and subsequent fields are files or arbitrary strings (admittedly newer versions of bash let you do more interesting things). A language like SQL on the other hand is (1) stateful, (2) needs to draw in external information about tables, columns, (3) also has to contend with name aliasing. Concretely: 1) Statefulness > SELECT * FROM X\t Could be completed with the rest of the table name (i.e. ""Xtensions""), "", [another table]"", or a SQL keyword ""FROM"", ""WHERE"". Typically when parsing the set of available options is represented by a state in a finite state machine compiled from a user-provided grammar. 2) External information > SELECT foo\t At this point in the query information comes from outside of the grammar: You need to query the database to figure out which fields begin with 'foo' 3) Aliasing: > SELECT foo.bar FROM myTable AS foo WHERE f\t The fact that 'foo' is a table name comes from inside the query. In other words, you need to partially parse the query to figure out how to parse the query completion. In short, we're looking to add tab completion to Mimir (http://mimirdb.info). Mimir is written in Java uses JLine3 to provide command line services (it has a completion widget). It would be nice if we could trash our existing SQL parser and replace it with something newer and more extensible as well."

"Mimir is a database system that automatically ""repairs"" messy data and allows users to query the fixed version. The gimmick is that we preserve the original values and other possible repairs of the data. Right now, we can do two things: - Query the most likely (Best Guess) repair - Generate a set of sampled repairs The goal of this project is to provide support for POSSIBLE and existential POSSIBLE queries. A POSSIBLE query: > SELECT POSSIBLE A, B, ... FROM ... ... computes and returns the full set of results that could possibly appear from SELECT A, B, ... FROM ... in any repair. Each tuple is emitted exactly once, regardless of how many possible worlds it appears in (although different tuples with the same values may be emitted). An optional CONF() function call emits the probability of that particular row appearing. The first goal would be to add support for POSSIBLE queries in Mimir. However, there are two limitations to solely POSSIBLE queries. First, POSSIBLE queries are slow (they require enumerating a very large state space). Second, in some cases (e.g. when a single problem admits an infinite or continuous domain of repairs), POSSIBLE queries may produce an infinite number of outputs. The second goal then would be to combine POSSIBLE and existential (EXISTS) queries. Concretely, an existential query > EXISTS SELECT * FROM R WHERE A = 5 ... returns simply true or false – true if the select query has at least one result, false otherwise. For example, the following query returns true if there is at least one row in R where A = 5. When combined with SELECT POSSIBLE queries, the resulting query essentially tests to see if there is any possible circumstance under which the query could return some value. Such queries are similar to why/why-not provenance, but not quite the same. Why/why-not provenance allows a wide variety of database modifications to serve as causal explanations (adding/removing tuples, changing values, etc...). For more background see: > https://github.com/UBOdin/mimir/wiki/Concepts > https://github.com/UBOdin/mimir/issues/178 > https://github.com/UBOdin/mimir/issues/245"

"The goal of the project is to develop a nightmare-style browser automation library for Kotlin. There are two things to explain here---one is what Kotlin is, and the other is what I mean by a nightmare-style browser automation library. First, Kotlin (https://kotlinlang.org) is a new language that has recently become one of the official languages of Android. It has some nice language features such as static typing, null safety, lambda functions, etc., but at the same time its grammar is concise. Because of that, it can be used for quick prototyping (comparable to python) as well as serious production development (comparable to Java). Now, a browser automation library allows you to interact with a browser programatically. For example, you can click web page buttons, fill out web forms, etc., all programmatically using a browser automation library. This is good for web app testing as well as browser task automation. There is (as far as I know) no browser automation library for Kotlin yet. Since Kotlin is a new language good for many things, I believe there has to be a browser automation library for Kotlin as well. The model we can follow is a browser automation library called nightmare (https://github.com/segmentio/nightmare). It provides a high-level API good enough for simple task automation and testing. There is a low-level browser automation library called Selenium Web Driver (http://www.seleniumhq.org/projects/webdriver/). This can be used as an underlying library. In summary, the goal is to develop a nightmare-style browser automation library for Kotlin."

A contactless power monitoring solution for installation in a residential electrical system. It would use current transformers to monitor the power usage on individual residential circuits (either at the breaker panel or elsewhere) and provide the data in some standardized form for local or cloud processing. There exist commercial solutions in this space, but they are generally either a) very expensive (typically not intended for residential use), b) proprietary cloud-based solutions with limited data protection or access to raw data, or c) complicated inference-based collection (using, e.g., only the incoming service lines for detection, not individual circuits), or even all of the above.

What might the team need to succeed?
A working knowledge of power systems (including, but not limited to, basic electrical principles and AC concepts such as power factor), data collection, electronics and software design. For "product" quality design, many standards will have to be engaged, and likely conformance testing from the FCC and an entity like UL. There are almost certainly NEC/NFPA concerns as well. The key for marketing of such a device would be ease of installation (either easy for a homeowner or inexpensive to hire an electrician) and a simple interface for understanding the collected data and its implications. Good engineering and HCI will be necessary.

What sort of team will it take?
EE, ME, CEng, and CS. The basic design is mostly EE/ME, and then CEng/CS can take the collected data as far as they want.
